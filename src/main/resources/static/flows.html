1. Seat hold — TTL soft hold → DB hard hold → paid

A two-stage lock: cheap, in-memory check-and-set survives a mobile network drop; the DB hard hold writes a payment deadline a scheduled sweeper later reaps.

State machine

stateDiagram-v2 [*] --> AVAILABLE AVAILABLE --> SOFT_HELD: soft-hold (Caffeine.putIfAbsent) SOFT_HELD --> AVAILABLE: TTL 5 min (Caffeine evict) SOFT_HELD --> HARD_HELD: hard-hold (consume token, SELECT FOR UPDATE) HARD_HELD --> SOLD: pay (within deadline) HARD_HELD --> AVAILABLE: deadline passed (HardHoldSweeper) HARD_HELD --> AVAILABLE: explicit release SOLD --> [*]

Walkthrough

  1. Pick seat — client POST /api/seats/{id}/soft-hold. SoftHoldStore.tryAcquire calls cache.asMap().putIfAbsent(seatId, hold). Atomic. Winner gets a token + expiresAt (5 min). Losers see 409 seat_held.
  2. Start checkout — client POST /hard-hold with the soft token. SoftHoldStore.consume validates and removes the in-memory hold; SELECT … FOR UPDATE locks the seat row; status flips AVAILABLE → HARD_HELD; a HardHold row is written with paymentDeadline = now + 120 s.
  3. Pay — POST /pay within the deadline: status flips HARD_HELD → SOLD, hold status becomes PAID.
  4. Auto-release — every 30 s HardHoldSweeper runs SeatService.sweepExpired: PENDING holds past the deadline are flipped to EXPIRED and the seat returns to AVAILABLE.
  5. Mobile drop — if the client never POSTs /hard-hold, the soft hold simply expires from Caffeine after 5 min. No DB pressure, no manual cleanup.

2. Calendar conflict — recurring rules + multi-timezone + pessimistic overlap

Rules are wall-clock, instants are UTC, overlap test runs under a pessimistic lock so two bookers cannot both win the same window.

Booking pipeline

flowchart TD A[Client posts
startInInviteeTz + tz + duration] --> B[CalendarService.book] B --> C[Resolve invitee ZoneId → Instant startUtc, endUtc] C --> D[Apply owner bufferMinutes on both sides] D --> E[BookingRepository.findOverlappingForUpdate
PESSIMISTIC_WRITE] E -->|overlap empty| F[INSERT Booking
status=CONFIRMED] E -->|overlap found| G[throw 409 slot_taken
with colliding invitee] F --> H[200 OK
return BookingView]

Walkthrough

  1. Recurring rules are wall-clock — RecurringRule stores LocalTime startLocal + dayOfWeek in the owner's IANA zone. "9:00 every Monday" stays 9:00 across DST shifts because we never freeze it to a UTC offset.
  2. Slot expansion — AvailabilityService.openSlots walks the date range, applies the rules, removes per-date RecurrenceException blockers, subtracts confirmed bookings (with owner buffer), and emits open slots as Instant pairs.
  3. Invitee submits local time + tz — POC accepts the invitee's wall-clock and their IANA zone. Server resolves to UTC: localStart.atZone(inviteeZone).toInstant().
  4. Buffer expansion — for the overlap query we widen by owner.bufferMinutes on both ends, so back-to-back bookings still respect the gap.
  5. Pessimistic overlap check — findOverlappingForUpdate runs start_a < end_b AND end_a > start_b under LockModeType.PESSIMISTIC_WRITE. Two clients arriving at the same millisecond serialize; loser sees 409 slot_taken.
  6. UTC always wins — DB stores Instant only; invitee timezone is kept as a label for display, never used for comparison.

3. Overbooking — probabilistic sell + two-pass bumping + compensation ledger

Sell beyond capacity; at boarding, prefer volunteers (cheaper to comp), then bump involuntary in reverse fare-rank.

Boarding flow

Walkthrough

  1. Sell ceiling = capacity × (1 + overbookFactor). OverbookingService.book locks the flight row (SELECT FOR UPDATE) and counts CONFIRMED bookings — if we're already at the ceiling, return 409 sold_out.
  2. At boarding, request lists noShowBookingIds. Those flip to NO_SHOW and don't compete for seats.
  3. Volunteer pass — survivors sorted by volunteerWillingness desc, then by lowest fare rank (BASIC_ECONOMY first). Anyone ≥ 30 willingness takes the voluntary-bonus ($200 default) and is bumped first. Pass stops when overflow hits 0 or willing volunteers run out.
  4. Involuntary pass — remaining sorted by reverse fare rank then most recent createdAt: cheapest classes go first, and within a class the latest sales bump first. Each pays involuntary-compensation ($800 default).
  5. Compensation ledger — every bump persists compensationPaid. The boarding response totals it so analytics can compare against the extra revenue earned by selling those overbooked tickets.
  6. Why this ordering — voluntary first because $200 < $800 per bump; reverse-fare second because the legal/contractual exposure is lowest on the cheapest fare classes; recent-first ensures loyalty/long-booked passengers stay onboard.

Tuning: if no-show rate is empirically 10%, then expected show-ups at ceiling 12 is 12 × 0.9 = 10.8. We over-board by ~0.8 on average — comp paid less than the revenue of the 2 extra tickets sold. That's the trade.
